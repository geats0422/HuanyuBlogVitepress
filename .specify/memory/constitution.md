# 真理之剑 项目章程

## 核心原则

### I. 内容优先

博客的核心价值在于高质量的内容分享。每篇文章必须基于真实实践经验，避免空洞的理论堆砌。

- 所有文章必须基于作者的亲身实践或深入研究
- 优先分享可操作的、能帮助读者解决实际问题的内容
- 技术文章应包含完整的实现步骤和关键代码/配置
- 避免转载或简单的摘要整理，除非有独特的分析视角

**理由**: 读者关注此博客是为了获取真实、可验证的实践经验，而非泛泛而谈的理论内容。

### II. 实践验证

分享的工具、方案或产品功能必须经过作者的实际验证。

- 技术方案必须已在作者的环境中成功运行
- 产品体验必须基于真实使用，而非营销文案复述
- 遇到的问题和坑点必须如实记录，这是最有价值的部分
- 如方案未完全验证，必须明确标注"实验性"或"进行中"

**理由**: 未经验验的内容可能误导读者，如实记录问题反而更具参考价值。

### III. 持续更新

技术领域快速迭代，文章应保持时效性。

- 重要文章应标注"最后更新"时间
- 当工具或平台发生重大变化时，应考虑更新相关文章
- 对于过时的内容，应添加废弃说明或重定向到新文章
- 建立"补充文章"机制，对原有内容进行扩展

**理由**: AI 和低代码平台发展迅速，过时的内容可能对读者造成困扰。

### IV. 简洁表达

内容应清晰易懂，避免不必要的术语和冗余。

- 使用简洁的语言表达复杂概念
- 技术术语应提供解释或上下文
- 文章结构应清晰：背景 → 过程 → 结果 → 总结
- 适当使用图表、代码块提升可读性

**理由**: 博客读者可能有不同的技术背景，过度专业的术语会降低内容的可访问性。

### V. 用户价值

每篇文章都应明确回答"读者能获得什么"。

- 文章开头应概述读者将学到的内容或解决的问题
- 优先分享能直接应用于读者项目的内容
- 避免自我记录式的流水账，除非对读者有启发价值
- 欢迎读者反馈，并将其作为内容改进的依据

**理由**: 博客的存在是为了分享和帮助他人，而非单纯的个人日记。

## 内容主题

博客内容聚焦于以下主题领域：

### 1. 低代码智能体搭建
- n8n、dify 等平台的实践教程
- 工作流设计和优化经验
- 常见问题和解决方案

### 2. 产品开发全流程
- 从想法到上线的完整记录
- 技术选型决策过程
- 开发中的挑战和应对

### 3. 细节补充
- 主体文章未尽的技术细节
- 实践中遇到的坑点和解决
- 读者的常见问题解答

### 4. AI 产品体验
- 新功能的实际使用评测
- 与同类产品的对比分析
- 适用场景和建议

## 技术规范

### 站点架构
- **静态站点生成器**: VitePress 1.6+
- **内容格式**: Markdown
- **配置语言**: TypeScript (.mts)
- **主语言**: 中文（内容）、英文（技术文档）

### 内容组织
- 文章存放在 `docs/` 目录
- 使用清晰的文件命名和目录结构
- 导航和侧边栏在 `docs/.vitepress/config.mts` 中配置

## 治理

### 修正流程
- 章程修改需经过充分讨论和文档化
- 重大变更（影响内容方向）需更新版本号
- 所有修改应在此文件顶部记录变更摘要

### 合规审查
- 新文章发布前应检查是否符合核心原则
- 定期审查现有文章的时效性和准确性
- 读者反馈应作为内容质量改进的重要参考

### 运行时指导
- 开发过程中参考 `CLAUDE.md` 获取技术指导
- 内容创作遵循本章程的原则约束
- 使用 Speckit 工作流管理功能开发任务
